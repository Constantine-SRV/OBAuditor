# OBAuditor — История изменений

## v20260422 (2026-04-23)

**Пересылка событий аудита в rsyslog (UDP)**

Новый класс `RsyslogSender` отправляет все новые события в rsyslog по UDP (RFC 3164)
в конце каждого прогона. Три типа событий с независимыми курсорами в таблице `rsyslog_cursor`:

- `login`  — новые строки `sessions` (курсор по `id`)
- `logoff` — закрытые сессии (курсор по паре `logoff_time + id`)
- `ddl`    — новые строки `ddl_dcl_audit_log` (курсор по `id`)

Курсор logoff использует пару `(logoff_time, id)` вместо просто `id`: сессия с малым `id`
может закрыться позже сессии с большим `id`, и курсор только по `id` пропустил бы её.

При недоступности rsyslog — ошибка в stderr, курсор не двигается, данные остаются
в таблицах и будут отправлены при следующем успешном прогоне.

Новые параметры конфига (`<Rsyslog>`):

```xml
<Rsyslog>
    <Host>192.168.55.200</Host>
    <Port>514</Port>
    <BatchSize>500</BatchSize>
</Rsyslog>
```

Пустой `<Host>` — пересылка отключена. Все новые записи отправляются за один цикл
запуска через несколько батчей подряд. Новая таблица `rsyslog_cursor` создаётся
автоматически при старте.

**Исправление фильтра DDL/DCL коллектора**

Глобальные исключения собственных служебных запросов сервиса вынесены на верхний
уровень `WHERE` (до блока `AND (...)`):

```sql
AND query_sql NOT LIKE '%INSERT IGNORE INTO admintools.ddl_dcl_audit_log%'
AND query_sql NOT LIKE '%UPDATE sessions SET logoff_time%'
AND query_sql NOT LIKE '%UPDATE sessions p JOIN sessions s%'
```

До этого исправления exclusion-паттерны находились внутри OR-блока и могли быть
обойдены двумя путями:

1. OceanBase JDBC добавляет `/* comment */` перед каждым запросом — паттерны без
   ведущего `%` не совпадали, и служебные запросы попадали в аудит.
2. Dynamic targets (`ddl_dcl_audit_targets`) матчили имена объектов прямо внутри
   текста нашего же INSERT IGNORE — query_sql коллектора содержит все LIKE-условия
   в своём WHERE-clause, включая названия аудируемых таблиц.

Вынос на верхний уровень решает обе проблемы: ни один OR-блок внутри, включая
dynamic targets, не может обойти глобальные исключения.

## v20260415 (2026-04-21)

**Кастомные объекты аудита**

Добавлена таблица `ddl_dcl_audit_targets`. Позволяет аудировать DML-операции над конкретными
таблицами, процедурами и вьюшками без перекомпиляции:

```sql
INSERT INTO admintools.ddl_dcl_audit_targets (tenant_id, db_name, object_name, description)
VALUES (1002, 'testdb', 'HISTORY_OPERATION', 'Аудит DML по таблице истории операций');
```

Изменения применяются при следующем запуске сервиса. Для отключения: `SET is_active = 0`.

**Аудит DELETE/UPDATE таблиц журналов**

В хардкод фильтра добавлен перехват `DELETE` и `UPDATE` по таблицам `admintools.sessions`
и `admintools.ddl_dcl_audit_log`. Служебные UPDATE сервиса (закрытие логоффов, reconciliation)
исключены автоматически.

## v20260409 (2026-04-09)

**DDL/DCL аудит из GV$OB_SQL_AUDIT**

Сбор DDL/DCL событий: CREATE/DROP/ALTER TABLE/INDEX/VIEW/DATABASE, GRANT, REVOKE, SET_PASSWORD,
CREATE USER, ALTER USER, lock_user(), unlock_user(). Режимы: основной (1) и резервный (2).

**Управление размером таблиц**

Плановое удаление старых строк по минуте часа. Разные минуты на разных узлах (0, 20, 40)
гарантируют удаление минимум раз в час.
